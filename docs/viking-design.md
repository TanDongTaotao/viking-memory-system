## 设计理念

### 核心灵感

Viking 记忆系统的设计灵感来源于**人类记忆的工作机制**：

> 人的记忆并非完美存储，而是会随着时间自然衰减。
> 
> - 短期记忆（工作记忆）：只能保存 7±2 个信息块
> - 长期记忆：通过复述和强化转化为长期记忆
> - 遗忘曲线：艾宾浩斯遗忘曲线显示信息会随时间指数衰减
> - 重要记忆：情绪强烈或重复记忆的内容更难忘

### 设计目标

1. **模拟人类记忆层级** - 从完整细节到抽象标签的自然过渡
2. **重要性感知** - 重要信息获得更高权重，遗忘更慢
3. **可恢复的遗忘** - 归档不是删除，搜索可触发回忆
4. **自动无需干预** - 自动化压缩和权重管理

### 对比传统方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| 完整保存 | 保留所有信息 | Token 成本高，检索困难 |
| 向量数据库 | 支持语义搜索 | 存储成本高，无法压缩 |
| **Viking** | **模拟人脑，智能压缩** | **实现复杂度较高** |

## 记忆模型

### 记忆生命周期

```
┌─────────────────────────────────────────────────────────────────────┐
│                        记忆生命周期                                  │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  完整细节 ──→ 轮廓摘要 ──→ 关键词 ──→ 极简标签 ──→ 归档(不删除)   │
│    │           │           │           │           │              │
│   0天         2天         8天        30天        90天+           │
│                                                                     │
│  [完整内容]  [核心轮廓]   [关键词]    [极简标签]   [可搜索回忆]     │
│                                                                     │
│                    ▲ 搜索触发回忆 ▲                                 │
└─────────────────────────────────────────────────────────────────────┘
```

### 层级定义

| 层级 | 时间范围 | 内容保留 | 清晰度 | 能否回忆 |
|------|----------|----------|--------|----------|
| L0-原始 | 0-1天 | 100% 完整细节 | 100% | 自动 |
| L1-轮廓 | 2-7天 | 核心轮廓/摘要 | ~70% | 自动 |
| L2-关键词 | 8-30天 | 关键信息 | ~30% | 搜索触发 |
| L3-标签 | 30-90天 | 极简标签 | ~10% | 搜索触发 |
| L4-归档 | 90天+ | 归档(不删除) | 0% | 搜索可回忆 ✓ |

### 元数据结构

```yaml
---
# 核心字段
id: mem_20260314_001           # 唯一标识符
title: "项目启动会议"           # 记忆标题

# 重要性管理
importance: high|medium|low   # 重要性等级
important: true|false         # 永不遗忘标记
tags: [project, meeting]      # 标签

# 时间追踪
created: 2026-03-14T10:30:00Z
last_access: 2026-03-14T14:22:00Z

# 访问统计
access_count: 5
access_history:               # 最近5次访问
  - 2026-03-14T14:22:00Z
  - 2026-03-14T11:15:00Z

# 保留策略
retention: forever|30|90|365
auto_archive: true

# 当前层级
current_layer: L0
level: 0
weight: 14.2
---
```

## 遗忘算法

### 权重计算公式

```
权重 = 重要性因子 × 时间衰减因子 × 访问增强因子

W = importance_factor × (1 / (days + 1)^0.3) × (access_count + 1)
```

### 重要性因子

| importance | importance_factor | 说明 |
|------------|-------------------|------|
| high | 3.0 | 核心记忆 |
| medium | 1.5 | 一般重要 |
| low | 0.5 | 次要信息 |
| important=true | 999 (max) | 永不遗忘 |

### 时间衰减曲线

```
权重变化 (重要性=medium, access_count=0)
│
14┤                                 *
12┤                              *
10┤                           *
 8┤                        *
 6┤                    *
 4┤                *
 2┤            *
 0┤__________*__________________________
   0    10    20    30    40    50    60 (天)
```

### 权重与层级映射

| 权重范围 | 建议层级 | 动作 |
|----------|----------|------|
| W >= 10 | L0 | 保留完整内容 |
| 5 <= W < 10 | L1 | 保留核心轮廓 |
| 2 <= W < 5 | L2 | 保留关键词 |
| 0.5 <= W < 2 | L3 | 仅极简标签 |
| W < 0.5 | L4 | 归档(搜索可回忆) |

### 权重更新时机

1. **访问时**: `access_count++`, `last_access = now()`
2. **写入时**: 新记忆默认权重 = importance_factor × 2
3. **每日cron**: 批量检查权重，触发压缩

## 压缩策略

### 压缩触发器

```python
TRIGGERS = {
    1:  'compress_to_contour',    # 1天 → 轮廓
    7:  'compress_to_keywords',   # 7天 → 关键词 (更新为8天)
    30: 'compress_to_tags',       # 30天 → 标签
    90: 'mark_for_archive',       # 90天 → 归档(不删除)
}
```

### LLM 压缩提示词

#### 轮廓压缩 (L0 → L1)

```
## 任务：生成记忆轮廓

原始记忆：
{{完整内容}}

要求：
1. 保留核心轮廓和关键要点（不超过150字）
2. 保留关键人物、日期、决策
3. 简化过程描述

输出格式：
## 轮廓
[你的轮廓]
```

#### 关键词提取 (L1 → L2)

```
## 任务：提取关键词

轮廓内容：
{{轮廓}}

要求：
1. 提取8-15个核心关键词
2. 使用逗号分隔
3. 优先选择：人物、项目、技术、日期、动作

输出格式：
### 关键词
关键词1, 关键词2, 关键词3, ...
```

#### 极简标签 (L2 → L3)

```
## 任务：提取极简标签

关键词内容：
{{关键词}}

要求：
1. 提取3-5个最核心的标签
2. 使用#号前缀
3. 只保留最关键的元素

输出格式：
### 标签
#标签1 #标签2 #标签3
```

## 回忆机制

### 搜索触发回忆流程

```
用户搜索 "项目启动"
    ↓
检索 L4 归档记忆（匹配关键词/标签）
    ↓
找到：#项目 #启动 #2026Q1
    ↓
┌─────────────────────────────────────┐
│  触发 LLM "回忆" 恢复               │
│                                     │
│  输入：关键词 + 时间上下文          │
│  输出：恢复完整/详细记忆           │
└─────────────────────────────────────┘
    ↓
展示给用户
```

### LLM 恢复提示词

```
## 任务：从归档关键词恢复记忆

归档关键词：{{关键词}}
记忆创建时间：{{created}}
记忆标题：{{标题}}

要求：
1. 基于提供的关键词和时间上下文
2. 尽可能恢复记忆的完整细节
3. 合理推断当时的情境、人物、决策
4. 标注哪些是"推断"内容

输出格式：
## 恢复的记忆
[LLM 恢复的完整内容]

## 推断说明
[哪些内容是 LLM 推断的]
```

### 回忆质量

| 现有内容 | 恢复质量 |
|----------|----------|
| L2 关键词 | 高 (可扩展) |
| L3 标签 | 中 (需推断) |
| 仅有 L4 归档 | 低 (完全推断) |

## 技术选型

### 存储方案

| 方案 | 适用场景 | 优点 | 缺点 |
|------|----------|------|------|
| Markdown 文件 | 个人/小团队 | 简单、可读、易编辑 | 搜索慢 |
| SQLite | 中等规模 | 索引快、事务支持 | 需额外学习 |
| 向量库 (Chroma) | 语义搜索 | 支持相似度检索 | 存储大 |

### LLM 服务

| 服务 | 模型 | 特点 |
|------|------|------|
| Ollama | glm-4-flash | 本地部署、免费 |
| OpenAI | gpt-4 | 效果好、成本高 |
| 智谱 | glm-4 | 中文优化、国内可用 |

### 性能优化

1. **批量压缩**: 每日cron批量处理，避免实时延迟
2. **异步索引**: 写入时异步更新向量索引
3. **缓存**: 热门记忆缓存在内存中
